1. System Overview

The Garbage Collection Management System is a full-stack web application built for property managers who collect monthly garbage fees from tenants. It replaces manual paper-based records with a centralised digital system.

Core Features

  • Buildings management — register and manage multiple properties
  • Customer management — track tenants per building with their room number and monthly fee
  • Payments tracking — record payments month by month with full history
  • M-Pesa STK Push — send payment prompts directly to a tenant's phone via Safaricom Daraja API
  • Manual reference entry — record payments made outside the system using an M-Pesa transaction code
  • Dashboard — at-a-glance statistics for the current month
  • Per-building reports — printable monthly collection summaries
  • Authentication — username/password login with session cookies; self-service first-run registration

Typical Workflow

  1. Admin logs in to the system.
  2. Admin adds buildings and registers customers under each building.
  3. At the start of the month, admin opens the Payments page, selects the current month, and sends STK Push requests to tenants with unpaid status.
  4. When a tenant pays, M-Pesa calls the callback URL and the system automatically marks the payment as paid.
  5. Alternatively, admin manually enters the M-Pesa reference code for a tenant who paid directly.
  6. Admin generates a building-level report to review collection status and prints it.

5. Authentication & Security

5.1 How It Works

Authentication uses a custom implementation with no external libraries. It relies on the browser's Web Crypto API (available in both Edge and Node.js runtimes).

  1. User submits username and password on /login.
  2. The server queries the users table, retrieves the stored PBKDF2 hash, and verifies it against the submitted password.
  3. On success, the server creates a signed session token (HMAC-SHA256) and sets it as an httpOnly cookie named session.
  4. On every subsequent request, middleware.ts verifies the cookie's signature. Invalid or missing tokens are redirected to /login.

5.2 Password Storage

Passwords are hashed using PBKDF2-SHA256 with 100,000 iterations. Each password has a unique 16-byte random salt. The stored value is: 

base64url(salt) : base64url(derivedKey)

Verification re-derives the key with the same salt and compares using a constant-time bitwise XOR to prevent timing attacks.

5.3 Session Tokens

The token format is payload.signature where payload is base64url-encoded JSON containing the username and issue timestamp. The signature is HMAC-SHA256 over the payload using the SESSION_SECRET environment variable.

5.4 Middleware — Route Protection

middleware.ts runs on the Next.js Edge runtime and protects all routes. Public paths that do not require authentication:

  • /login
  • /register
  • /api/auth/login
  • /api/auth/register
  • /api/mpesa/callback — must remain public for Safaricom to reach it

5.5 First-Run Registration

The POST /api/auth/register endpoint checks whether any users exist in the database. If the users table is empty, registration is allowed and the first account is created. All subsequent registration attempts return 403 Forbidden. This means only the first admin can register — additional accounts must be created directly in the database.

5.6 Fallback Credentials

If the users table is empty (e.g. before first registration), the login route falls back to the ADMIN_USERNAME and ADMIN_PASSWORD environment variables. This allows access immediately after deployment before the database is set up.

6.5 M-Pesa Endpoints

POST /api/mpesa/stk-push

Triggers an STK Push (payment prompt) to the customer's phone and creates a pending payment record.

Body FieldTypeNotes
customer_idnumberRequired
monthnumberRequired
yearnumberRequired

The handler fetches the customer record, normalises the phone number to 2547XXXXXXXX format, calls initiateStkPush() from lib/mpesa.ts, then inserts a payment row with status='unpaid' using ON CONFLICT DO NOTHING.

Success: 200, returns Daraja's StkPushResult containing CheckoutRequestID.

POST /api/mpesa/callback

Webhook endpoint called by Safaricom after the customer enters their PIN. This route is intentionally not protected by authentication — it must be publicly accessible for Safaricom's servers to reach it.

On a successful payment (ResultCode === 0), it extracts MpesaReceiptNumber, Amount, and PhoneNumber from the callback metadata and updates the most recent pending stk_push payment for that phone number to status='paid'.

Always responds with {"ResultCode": 0, "ResultDesc": "Accepted"} to satisfy Safaricom's expectation.

7. Frontend Pages

7.1 Login — /login

Clean centered card with username/password fields and a show/hide password toggle. On successful authentication, redirects to /dashboard. Includes a link to the registration page.

7.2 Register — /register

First-run account setup page. Features:

  • Password strength indicator (Weak / Fair / Good / Strong) based on length, casing, digits, and special characters
  • Confirm password field with a live checkmark when passwords match
  • Automatically logs in and redirects to dashboard on success
  • If a user already exists in the DB, the API returns 403 and a clear error message is shown

7.3 Dashboard — /dashboard

A Next.js Server Component. Queries the database directly (no HTTP fetch) and renders six StatCard components in a responsive 3-column grid:

  • Total Buildings
  • Total Customers
  • Collected This Month (KES)
  • Expected This Month (KES)
  • Outstanding (KES) — difference between expected and collected
  • Customers Paid — count with fraction of total

7.4 Buildings — /buildings

Client component. Displays all buildings as cards in a 2-column grid. Each card shows the building name, location, and a link to view customers filtered to that building. Supports Add, Edit, and Delete via modal forms.

7.5 Customers — /customers

Client component. Displays a table of all customers with their current month's payment status. Can be filtered by building. Supports Add, Edit, and Delete. Links from the Buildings page pre-filter to a specific building via the ?building_id query parameter.

7.6 Payments — /payments

The most feature-rich page. Displays a table of all customers for the selected month/year/building, with each customer's payment details.

Two actions are available per customer:

  • STK Push — opens a confirmation modal and, on confirm, calls POST /api/mpesa/stk-push. The customer receives a payment prompt on their phone within seconds.
  • Enter Reference — opens a form modal to record a payment that was made outside the system. The admin enters the M-Pesa transaction code, amount, and phone number.

7.7 Reports — /reports

Allows the admin to select a building, month, and year to generate a collection report. Displays a summary row of stats (KES collected, outstanding, count paid/unpaid) and a detailed table per customer.

A Print button triggers window.print(). The UI uses print:hidden Tailwind classes to hide controls and show only the report table when printing.

8. Shared Components

AppShell — components/AppShell.tsx

Client component that wraps all authenticated pages. On screens wider than 768px (Tailwind md breakpoint), the sidebar is always visible. On mobile, the sidebar is hidden off-screen and a top bar with a hamburger menu button slides it in as an overlay with a dark backdrop.

The login page is detected by usePathname() and rendered without the shell (no sidebar or top bar).

Sidebar — components/Sidebar.tsx

Dark slate (bg-slate-900) navigation sidebar. Active links are highlighted with bg-slate-700. The active route is detected using exact match or startsWith(href + '/') to avoid false positives. Includes a Sign Out button at the bottom that calls POST /api/auth/logout and redirects to /login.

StatCard — components/StatCard.tsx

White card with a coloured icon badge, a large bold value, a label, and an optional sub-text. Accepts a lucide-react icon, icon colour class, and icon background class as props.

Modal — components/Modal.tsx

Accessible modal dialog. Closes on Escape key press or clicking the dark backdrop. Renders in a portal-like fixed overlay with z-50.

StatusBadge — components/StatusBadge.tsx

Pill badge showing payment status. Three variants: paid (green), unpaid (red), partial (yellow).

9. M-Pesa Integration

9.1 Overview

The system integrates with Safaricom's Daraja API v1 using the Lipa Na M-Pesa Online (STK Push) product. This product allows a business to initiate a payment prompt on a customer's phone — the customer only needs to enter their M-Pesa PIN to complete the payment.

Note: The Daraja sandbox uses test credentials. Real money is never moved in sandbox mode. To go live, update MPESA_ENV=production in .env.local and use production credentials from the Daraja portal.

9.2 Authentication Flow

  1. The system calls GET /oauth/v1/generate?grant_type=client_credentials on the Daraja base URL with Basic Auth (Consumer Key:Consumer Secret in base64).
  2. Daraja returns a short-lived access_token.
  3. The access token is used as a Bearer token in the STK Push request.

9.3 STK Push Request

The STK Push payload sent to /mpesa/stkpush/v1/processrequest:

FieldValue
BusinessShortCodeMPESA_SHORTCODE env var (sandbox: 174379)
Passwordbase64(ShortCode + PassKey + Timestamp)
TimestampYYYYMMDDHHmmss — current time
TransactionTypeCustomerPayBillOnline
AmountCustomer's monthly fee (rounded up with Math.ceil)
PartyACustomer's phone (2547XXXXXXXX format)
PartyBBusiness shortcode
PhoneNumberCustomer's phone
CallBackURLMPESA_CALLBACK_URL env var
AccountReferenceGC-{room_number}
TransactionDesc"Garbage fee {month} {year}"

9.4 Callback Handling

When the customer completes payment, Safaricom sends a POST request to MPESA_CALLBACK_URL with the following structure: 

{
  "Body": {
    "stkCallback": {
      "MerchantRequestID": "...",
      "CheckoutRequestID": "...",
      "ResultCode": 0,
      "ResultDesc": "The service request is processed successfully.",
      "CallbackMetadata": {
        "Item": [
          { "Name": "Amount", "Value": 500 },
          { "Name": "MpesaReceiptNumber", "Value": "QK7BX3RT2" },
          { "Name": "PhoneNumber", "Value": 254712345678 }
        ]
      }
    }
  }
}

The callback route extracts the receipt number, amount, and phone number. It finds the most recent pending stk_push payment for that phone and updates it to status='paid' with the M-Pesa reference.

A ResultCode other than 0 means the customer cancelled or the payment failed — in this case the payment record remains as unpaid.

9.5 Important: ngrok for Local Development

Safaricom's servers cannot reach localhost. During development, use ngrok to create a public tunnel: 

ngrok http 3000

Copy the generated HTTPS URL and set it in .env.local:

MPESA_CALLBACK_URL=https://xxxx.ngrok-free.app/api/mpesa/callback

Restart the Next.js dev server whenever the ngrok URL changes (free plan generates a new URL on each restart).

11. Deployment Guide

The recommended free hosting stack is Vercel (application) + Neon (PostgreSQL).

Step 1 — Push code to GitHub

Create a GitHub repository and push the project. Ensure .env.local is in .gitignore (it is by default in Next.js projects).

Step 2 — Create a Neon database

  1. Sign up at neon.tech
  2. Create a new project and copy the DATABASE_URL connection string
  3. Open the Neon SQL editor and paste the contents of lib/schema.sql to create the tables

Step 3 — Deploy to Vercel

  1. Sign up at vercel.com
  2. Click "Add New Project" and import your GitHub repository
  3. Vercel will auto-detect Next.js — no build config needed
  4. In the Environment Variables section, add all variables from the table above:
    • Use the Neon DATABASE_URL
    • Set NEXT_PUBLIC_BASE_URL to https://your-app.vercel.app
    • Generate a strong random SESSION_SECRET
    • Set your Daraja production credentials (or sandbox for testing)
    • Set MPESA_CALLBACK_URL to https://your-app.vercel.app/api/mpesa/callback
  5. Click Deploy

Step 4 — Create admin account

Visit https://your-app.vercel.app/register to create the first admin account. This is only possible once — subsequent visits will show "Registration is closed".

SSL Note

Neon requires SSL. In lib/db.ts, the production SSL setting is currently rejectUnauthorized: false. For production, change this to true (Neon's CA is trusted by Node.js by default).

12. Known Limitations & Future Work

#IssueImpactSuggested Fix
1 No M-Pesa callback signature verification An attacker who knows the callback URL could POST a fake paid status Append a secret token to the callback URL and verify it on every inbound request
2 CheckoutRequestID not stored If a customer has multiple pending months, the callback may update the wrong record Add a checkout_request_id column to payments and match on it in the callback
3 STK Push is sent before the DB record is created If the INSERT fails, no record exists to match the callback against Insert the pending record first, then trigger the push; delete the record if the push fails
4 No try/catch on API routes DB errors expose stack traces in the HTTP response Wrap each route in a try/catch that returns a generic 500 JSON response
5 paid_at set for unpaid records paid_at is meaningless for pending payments Only set paid_at when status transitions to 'paid'
6 Single admin account only No way to add more users via the UI Add a Users management page accessible only when logged in
7 Reports include partial payments in numUnpaid Summary counts are slightly inaccurate when partial payments exist Count partial as its own category in the summary
